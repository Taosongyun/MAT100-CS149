01Why We Care

Sorting is one of the most fundamental tasks in computing — organizing a list, ranking candidates, ordering search results. But there are dozens of sorting algorithms. How do we know which one is best? And is there a limit to how fast any sorting algorithm can be?

This week we answer those questions using ideas from Week 1. The key insight is this:

The Central Idea Sorting an unknown list of n elements means figuring out which of the n! possible orderings you have. Every comparison gives you at most 1 bit of information. So you need at least log₂(n!) comparisons — no matter how clever your algorithm is.

This connects directly to Week 1: permutations aren't just abstract math objects — they are the state space of every sorting problem.

02Definitions

What is Sorting?

Definition — Sorting Given a sequence of n elements a₁, a₂, …, aₙ with a comparison operation (<), sorting means rearranging them into a sequence aπ(1) ≤ aπ(2) ≤ … ≤ aπ(n) for some permutation π.

A comparison-based sorting algorithm learns about the input only by asking questions of the form "is aᵢ < aⱼ?" Each such question is called a comparison.
Definition — How to Prove a Sequence is Sorted To prove a sequence b₁, b₂, …, bₙ is sorted, you must verify two things:
  1. Order: b₁ ≤ b₂ ≤ … ≤ bₙ — each element is ≤ the next.
  2. Same elements: {b₁,…,bₙ} = {a₁,…,aₙ} as multisets — no elements were lost or added.
Both conditions are necessary. A sequence of all zeros is ordered but not a valid sort of [3,1,2].

Sorting as Navigating Permutation Space

Here is the crucial link between sorting and permutations from Week 1.

Key Connection When you start sorting, you don't know the order of the input. There are n! possible orderings — that is, n! permutations of {1,…,n}, each a possible "true ranking" of your input.

Each comparison you make splits the remaining possibilities:
  • Before any comparison: n! possible orderings consistent with what you know.
  • After one comparison (is aᵢ < aⱼ?): at most n!/2 remain (roughly).
  • You are done when exactly 1 ordering remains.
This means sorting is equivalent to identifying an unknown permutation by making binary queries.

The maximum number of comparisons any algorithm can ever need is C(n,2) = n(n−1)/2 — just compare every pair. This is always sufficient because knowing every pairwise comparison uniquely determines the sorted order.

Big O, Ω, and Θ Notation

We need a precise language for describing how fast algorithms grow. Think of T(n) as the number of comparisons an algorithm makes on an input of size n.

Definition — Big O (Upper Bound) We say T(n) = O(f(n)) if T(n) grows at most as fast as f(n). Formally: there exist constants c > 0 and N such that T(n) ≤ c · f(n) for all n > N.

Plain English: "T(n) is eventually bounded above by a constant multiple of f(n)."
Example: Insertion sort worst case: T(n) = n(n−1)/2 = O(n²).
Definition — Ω (Lower Bound) We say T(n) = Ω(f(n)) if T(n) grows at least as fast as f(n). Formally: there exist constants c > 0 and N such that T(n) ≥ c · f(n) for all n > N.

Plain English: "T(n) is eventually bounded below by a constant multiple of f(n)."
Example: Any sorting algorithm: T(n) = Ω(n log n) in the worst case.
Definition — Θ (Tight Bound) We say T(n) = Θ(f(n)) if T(n) is both O(f(n)) and Ω(f(n)). That is:
Θ(f(n)) = O(f(n)) ∩ Ω(f(n))
Plain English: "T(n) grows at exactly the same rate as f(n), up to constant factors."
Example: Merge sort: T(n) = Θ(n log n) — both the best and worst cases are n log n.
How to remember which is which
  • O = "Oh, it's at most this bad" → upper bound → ceiling
  • Ω = "Omega, it's at least this hard" → lower bound → floor
  • Θ = "Theta, it's exactly this" → tight bound → sandwiched
A useful analogy: if T(n) is your salary, O is your maximum possible pay, Ω is your minimum guaranteed pay, and Θ means they're the same (fixed salary).

The Information-Theoretic Lower Bound

Theorem — Lower Bound on Comparisons Any correct comparison-based sorting algorithm must make at least ⌈log₂(n!)⌉ comparisons in the worst case.
Proof — Decision Tree Argument

Model any sorting algorithm as a binary decision tree:

  • Each internal node is a comparison: "is aᵢ < aⱼ?"
  • Each branch is an outcome: Yes (left) or No (right)
  • Each leaf is a conclusion: "the sorted order is π"

For the algorithm to be correct, it must be able to produce any of the n! possible sorted orders. So the tree must have at least n! leaves.

A binary tree of depth d has at most 2ᵈ leaves. So we need 2ᵈ ≥ n!, which gives d ≥ log₂(n!), i.e. d ≥ ⌈log₂(n!)⌉.

The depth of the tree = the number of comparisons in the worst case. ∎

Concrete values
  • n=4: 4!=24, so at least ⌈log₂(24)⌉ = 5 comparisons needed.
  • n=8: 8!=40320, so at least ⌈log₂(40320)⌉ = 16 comparisons needed.
  • n=16: 16! is huge, so at least ⌈log₂(16!)⌉ = 45 comparisons needed.

04The Five Sorting Algorithms

For each algorithm we give: the idea in plain English, the best case walkthrough, the worst case walkthrough, and the comparison counts.

Insertion Sort

The Idea Build a sorted portion from left to right. For each new element, insert it into its correct position in the already-sorted portion by shifting larger elements one step right.

Think of it like sorting a hand of playing cards: you pick up cards one at a time and slide each into the right spot among the ones you're already holding.
🟢 Best Case — already sorted input [1, 2, 3, 4] 3 comparisons

Each new element only needs one comparison — it's already larger than the last element in the sorted portion.

Start: sorted=[1], unsorted=[2,3,4] Insert 2: compare 2 vs 1 → 2>1, place after. sorted=[1,2] → 1 comparison
Insert 3: compare 3 vs 2 → 3>2, place after. sorted=[1,2,3] → 1 comparison
Insert 4: compare 4 vs 3 → 4>3, place after. sorted=[1,2,3,4] → 1 comparison
Total: 3 comparisons = n−1 ✓
🔴 Worst Case — reverse sorted input [4, 3, 2, 1] 6 comparisons

Each new element must travel all the way to the front, comparing with every element in the sorted portion.

Start: sorted=[4], unsorted=[3,2,1] Insert 3: compare 3 vs 4 → 3<4, shift 4 right. sorted=[3,4] → 1 comparison
Insert 2: compare 2 vs 4 → shift, compare 2 vs 3 → shift. sorted=[2,3,4] → 2 comparisons
Insert 1: compare 1 vs 4 → shift, 1 vs 3 → shift, 1 vs 2 → shift. sorted=[1,2,3,4] → 3 comparisons
Total: 1+2+3 = 6 comparisons = n(n−1)/2 ✓
HW April 6 — Fixing Insertion Sort The problem with insertion sort's worst case is that finding the right position takes O(n) time even though the sorted portion is already sorted. We can use binary search to find the insertion position in O(log n) comparisons instead of O(n). This reduces the comparison count to O(n log n) worst case — while keeping the spirit of "insert into sorted portion."

With binary search to find position: for inserting the k-th element into a sorted list of k−1 elements, we need at most ⌈log₂(k)⌉ comparisons. Total: ∑_{k=2}^{n} ⌈log₂(k)⌉ = O(n log n).

Selection Sort

The Idea Repeatedly select the minimum of the unsorted portion and swap it to the front. After k passes, the first k elements are sorted and in their final positions.

Unlike insertion sort, selection sort always makes the same number of comparisons regardless of the input — it must scan the entire unsorted portion to find the minimum every time.
🟢 Best & Worst Case (same!) — [3, 1, 4, 2] always 6 comparisons

Selection sort always scans everything — it cannot stop early.

Pass 1: find min of [3,1,4,2] 3 vs 1 → min=1; 1 vs 4 → min=1; 1 vs 2 → min=1. Swap 1 to front. [1,3,4,2] → 3 comps
Pass 2: find min of [3,4,2] 3 vs 4 → min=3; 3 vs 2 → min=2. Swap 2. [1,2,4,3] → 2 comps
Pass 3: find min of [4,3] 4 vs 3 → min=3. Swap 3. [1,2,3,4] → 1 comp
Total: 3+2+1 = 6 comparisons = n(n−1)/2 ✓ (always)

Bubble Sort

The Idea Repeatedly scan through the list, swapping adjacent elements that are out of order. After each full pass, the largest unsorted element "bubbles up" to its correct position at the end.

With early stopping: if a full pass makes no swaps, the list is already sorted — stop immediately. This gives bubble sort a good best case.
🟢 Best Case — already sorted [1, 2, 3, 4] 3 comparisons
Pass 1: compare all adjacent pairs 1 vs 2 → ok; 2 vs 3 → ok; 3 vs 4 → ok. No swaps → STOP. [1,2,3,4] → 3 comps
Total: 3 comparisons = n−1 ✓
🔴 Worst Case — reverse sorted [4, 3, 2, 1] 6 comparisons
Pass 1: scan [4,3,2,1] 4 vs 3 → swap; 4 vs 2 → swap; 4 vs 1 → swap. Result: [3,2,1,4] → 3 comps
Pass 2: scan [3,2,1] 3 vs 2 → swap; 3 vs 1 → swap. Result: [2,1,3,4] → 2 comps
Pass 3: scan [2,1] 2 vs 1 → swap. Result: [1,2,3,4] → 1 comp
Total: 3+2+1 = 6 comparisons = n(n−1)/2 ✓

Merge Sort

The Idea Divide and conquer: split the list in half, recursively sort each half, then merge the two sorted halves. Merging two sorted lists is efficient — just repeatedly take the smaller of the two front elements.

Merge sort always splits evenly (for n = 2^k) and always recurses fully, so its comparison count doesn't vary much with the input.
🟢 Best Case — [1, 2, 3, 4] (already sorted) 4 comparisons

Even on a sorted input, merge sort does all its recursive splits. But each merge is cheap — the smaller list exhausts quickly.

Split: [1,2] and [3,4] Split further: [1],[2],[3],[4]
Merge [1] + [2]: 1 vs 2 → take 1, append 2. Result: [1,2] → 1 comp
Merge [3] + [4]: 3 vs 4 → take 3, append 4. Result: [3,4] → 1 comp
Merge [1,2] + [3,4]: 1 vs 3 → take 1; 2 vs 3 → take 2; append 3,4. Result: [1,2,3,4] → 2 comps
Total: 1+1+2 = 4 comparisons ✓
🔴 Worst Case — [1, 3, 2, 4] (maximally interleaved) 5 comparisons

The worst case for merge sort occurs when the two halves interleave perfectly — neither exhausts early, so every element is compared.

Split: [1,3] and [2,4]. Split further: [1],[3],[2],[4]
Merge [1]+[3]: 1 vs 3 → take 1, append 3. [1,3] → 1 comp
Merge [2]+[4]: 2 vs 4 → take 2, append 4. [2,4] → 1 comp
Merge [1,3]+[2,4]: 1 vs 2 → take 1; 3 vs 2 → take 2; 3 vs 4 → take 3; append 4. [1,2,3,4] → 3 comps (every element compared!)
Total: 1+1+3 = 5 comparisons ✓
Why merge sort's best and worst cases are so close Unlike insertion or bubble sort, merge sort can't "skip work" on nice inputs. It always splits fully and always merges. The only variation is whether each merge step takes k or 2k−1 comparisons. For n=4: best=4, worst=5. For n=16: best=32, worst=49.

Quick Sort

The Idea Pick a pivot element. Partition the list into: elements smaller than the pivot (left), and elements larger (right). Recursively sort each side. The pivot ends up in its final correct position after each partition step.

The efficiency depends entirely on pivot choice: a good pivot splits the list roughly in half (like merge sort). A bad pivot — always picking the smallest or largest — degenerates to O(n²).
🟢 Best Case — balanced splits [2, 1, 4, 3] (pivot=first element) 4 comparisons
Pivot = 2. Compare all others vs 2: 1 vs 2 → left; 4 vs 2 → right; 3 vs 2 → right. Partition: [1] | 2 | [4,3] → 3 comps
Left [1]: already single element, done. Right [4,3]: pivot=4. 3 vs 4 → left. Partition: [] | 4 | — wait, pivot is 4 so [3] | 4 | [] → 1 comp
Total: 3+1 = 4 comparisons ✓
🔴 Worst Case — already sorted [1, 2, 3, 4] (pivot=first element) 6 comparisons

When the pivot is always the smallest element, the "left" partition is always empty. This is the worst case — the recursion has depth n−1 instead of log n.

Pivot=1. Compare 2,3,4 vs 1: all go right. [] | 1 | [2,3,4] → 3 comps
Pivot=2. Compare 3,4 vs 2: all go right. [] | 2 | [3,4] → 2 comps
Pivot=3. Compare 4 vs 3: goes right. [] | 3 | [4] → 1 comp
Total: 3+2+1 = 6 = n(n−1)/2 comparisons ✓
Quick Sort Gotcha Quick sort's worst case occurs on already sorted or reverse sorted input when using a naive pivot (first or last element). This is the opposite of insertion sort! In practice, quick sort uses randomized pivot selection or the "median of three" strategy to avoid this.

05The Coin Weighing Problem

The Setup

Problem Statement You have n coins. One is counterfeit and is either lighter or heavier than the genuine coins (you don't know which). You have a balance scale that tells you which side is heavier, or if they're equal. You may use exactly k weighings. Identify the counterfeit coin and whether it is lighter or heavier.

Connection to Decision Trees

The Parallel with Sorting This is the same decision tree argument as sorting, but with base 3 instead of base 2:
  • Each weighing has 3 outcomes: left heavy, right heavy, balanced.
  • k weighings → decision tree of depth k → at most 3ᵏ leaves.
  • With n coins, each either lighter or heavier: 2n possible outcomes.
  • Need: 3ᵏ ≥ 2n, i.e., n ≤ 3ᵏ/2.
For k=3: 3³=27, so we need 2n ≤ 27, i.e., n ≤ 13. But — as we'll see — the actual bound is tighter: n ≤ 12.

Walkthrough: 12 Coins, 3 Weighings

Step-by-step strategy

Label coins 1–12. Strategy: weigh 4 vs 4 first, set 4 aside.

Weighing 1: Put coins {1,2,3,4} on the left, {5,6,7,8} on the right, {9,10,11,12} aside.

  • Each of the 3 outcomes leaves exactly 8 possibilities:
    • Left heavy: one of {1,2,3,4} is heavy OR one of {5,6,7,8} is light → 8 cases
    • Right heavy: one of {5,6,7,8} is heavy OR one of {1,2,3,4} is light → 8 cases
    • Balanced: counterfeit is in {9,10,11,12}, light or heavy → 8 cases
  • 2 more weighings → 3² = 9 ≥ 8 possibilities. ✓

Weighing 2 (suppose Weighing 1 was balanced): The counterfeit is in {9,10,11,12}. Weigh {9,10} vs {11,1} (using coin 1 as a known genuine coin).

  • Balanced: counterfeit is coin 12. One more weighing: 12 vs any genuine → determine lighter or heavier.
  • Left heavy: coin 9 or 10 is heavy, or coin 11 is light → 3 cases remain.
  • Right heavy: coin 11 is heavy, or coin 9 or 10 is light → 3 cases remain.

Weighing 3 (suppose left heavy from Weighing 2): Possibilities: {9 heavy, 10 heavy, 11 light}. Weigh 9 vs 10.

  • Left heavy: coin 9 is the counterfeit and it's heavy. ✓
  • Right heavy: coin 10 is the counterfeit and it's heavy. ✓
  • Balanced: coin 11 is the counterfeit and it's light. ✓

The same logic works for all other branches of Weighing 1. The full strategy resolves all 24 cases in exactly 3 weighings.

Why 13 Coins is Impossible (HW April 8)

Proof that 13 coins cannot be solved in 3 weighings

We must show that no matter how we design the first weighing, some branch leaves more than 3² = 9 unresolved possibilities.

In the first weighing, suppose we put k coins on each side and 13−2k aside. The three branches have:

  • Left heavy: 2k possibilities (k left-heavy coins + k right-light coins)
  • Right heavy: 2k possibilities
  • Balanced: 2(13−2k) possibilities

For the algorithm to succeed, all three branches must have at most 9 possibilities. Check every valid k:

No value of k works. The best attempt is k=4 (weigh 4 vs 4, set 5 aside), but the balanced branch leaves 10 possibilities — one too many for 2 remaining weighings. Therefore, 13 coins is impossible. ∎

Why 14 coins is also impossible (easier argument) With 14 coins: 2 × 14 = 28 possible outcomes, but 3³ = 27 leaves. Even in the best possible case, the decision tree doesn't have enough leaves. No strategy can work. ∎

Practice Problems — Sorting & Information Theory
Q1What is the information-theoretic lower bound on comparisons for sorting n=5 elements? Show your work.▼
Solution
Step 1: Count the number of possible orderings. 5! = 5×4×3×2×1 = 120.
Step 2: A binary decision tree needs at least 120 leaves. A tree of depth d has at most 2ᵈ leaves. We need 2ᵈ ≥ 120.
Step 3: Find the smallest d: 2⁶ = 64 < 120, 2⁷ = 128 ≥ 120. So d = 7.

Lower bound = ⌈log₂(120)⌉ = 7 comparisons

Q2T(n) = 3n² + 2n + 1. Show that T(n) = O(n²) by finding explicit constants c and N.▼
Solution

We need to find c > 0 and N such that 3n² + 2n + 1 ≤ c·n² for all n > N.

Strategy: bound the smaller terms. For n ≥ 1: 2n ≤ 2n² and 1 ≤ n². So:
3n² + 2n + 1 ≤ 3n² + 2n² + n² = 6n²

c = 6, N = 1 works: T(n) ≤ 6n² for all n ≥ 1

Many other choices of c and N are valid — the constants just need to exist.

Q3Trace insertion sort on [2, 4, 1, 3]. Count the exact number of comparisons at each step and give the total.▼
Solution

σ(1)=2, σ(2)=4, σ(3)=1, σ(4)=3. Sorted portion starts as [2].

Insert 4: Compare 4 vs 2 → 4 > 2, place after. Sorted: [2,4]. 1 comparison.
Insert 1: Compare 1 vs 4 → 1 < 4, shift. Compare 1 vs 2 → 1 < 2, shift. Place 1 at front. Sorted: [1,2,4]. 2 comparisons.
Insert 3: Compare 3 vs 4 → 3 < 4, shift. Compare 3 vs 2 → 3 > 2, place after 2. Sorted: [1,2,3,4]. 2 comparisons.

Total: 1 + 2 + 2 = 5 comparisons

Q4Trace merge sort on [3, 1, 4, 2]. Show all splits and merges, and count comparisons at each merge step.▼
Solution
Split: [3,1,4,2] → [3,1] and [4,2] → [3],[1],[4],[2]
Merge [3]+[1]: 3 vs 1 → take 1, append 3. Result: [1,3]. 1 comparison.
Merge [4]+[2]: 4 vs 2 → take 2, append 4. Result: [2,4]. 1 comparison.
Merge [1,3]+[2,4]:
1 vs 2 → take 1.
3 vs 2 → take 2.
3 vs 4 → take 3.
Append 4.
Result: [1,2,3,4]. 3 comparisons.

Total: 1 + 1 + 3 = 5 comparisons

Q5Selection sort always makes exactly n(n−1)/2 comparisons. Prove this.▼
Solution

In selection sort, pass i (for i = 1, 2, \ldots, n−1) finds the minimum of the remaining n−i+1 unsorted elements. To find the minimum of a list of m elements, you must compare all m elements — there is no early stopping.

Pass 1: scan n elements, make n−1 comparisons to find minimum.
Pass 2: scan n−1 elements, make n−2 comparisons.
Pass k: make n−k comparisons.
Total: (n−1) + (n−2) + \cdots + 1 = \sum_{k=1}^{n-1} k = \frac{n(n-1)}{2}.

This does not depend on the input — every element in the unsorted portion must be examined to guarantee we found the minimum. ∎

Total comparisons = n(n−1)/2, always

Q6For n=16, the lower bound is 45 comparisons. Merge sort's worst case is 49. How far is this above the lower bound, in percentage terms? What does this tell us?▼
Solution
Compute excess: 49 − 45 = 4 extra comparisons.
Percentage above lower bound: 4/45 × 100 ≈ 8.9\%.

Merge sort's worst case is only about 9% above the theoretical minimum for n=16. This tells us merge sort is nearly optimal — it's essentially as efficient as any comparison-based sorting algorithm could possibly be. As n grows, merge sort tracks the lower bound Θ(n log n) exactly.

Compare to selection sort: 120 comparisons vs 45 minimum — that's 75/45 ≈ 167\% above the lower bound. Selection sort wastes dramatically more work.

Merge sort worst case is ~9% above optimal; selection sort is ~167% above optimal

Q7With a balance scale and 4 weighings, what is the maximum number of coins you can test (one counterfeit, lighter or heavier)? Use the formula and verify the logic.▼
Solution
Apply the formula: max coins = (3^k − 3)/2 with k = 4:
(3^4 − 3)/2 = (81 − 3)/2 = 78/2 = 39 coins.
Verify the logic: With 39 coins and 4 weighings:
First weighing: put 13 coins on each side, 13 aside.
Each branch (left heavy, right heavy, balanced) leaves at most 2×13 = 26 or 2×13 = 26 possibilities.
With 3 remaining weighings: (3^3−3)/2 = 12... wait, that's for 3 weighings.
Actually: 3 remaining weighings can resolve (3^3−3)/2 = 12... no.
More carefully: 3 remaining weighings give 3^3 = 27 ≥ 26 leaves. ✓

Maximum = 39 coins with 4 weighings

Q8Quick sort with a first-element pivot is applied to [3,1,4,1,5,9,2,6]. (a) What is the first partition step? (b) Why is this input not the worst case?▼
Solution
(a) First partition step: Pivot = 3 (first element). Compare every other element vs 3:
1 < 3 → left; 4 > 3 → right; 1 < 3 → left; 5 > 3 → right; 9 > 3 → right; 2 < 3 → left; 6 > 3 → right.
Result: [1,1,2] | 3 | [4,5,9,6]. That's 7 comparisons.
Left subarray has 3 elements, right has 4 — a reasonably balanced split.
(b) Why not worst case: The worst case occurs when the pivot is always the smallest (or largest) element, creating a partition of [empty | pivot | everything else]. Here the pivot 3 splits the 8 elements into 3 and 4 — close to the ideal 4-4 split. Worst case would be input [1,2,3,4,5,6,7,8] where pivot=1 always lands first and creates maximally unbalanced splits.

(a) [1,1,2] | 3 | [4,5,9,6], 7 comparisons. (b) Not worst case — split is balanced.

HW — Due Monday April 6

Problem: Fix insertion sort — keeping its spirit — so its worst-case running time is better than Θ(n²).

Answer: Use binary search to find the insertion position. When inserting the k-th element into a sorted list of k−1 elements, binary search finds the correct position in ⌈log₂(k)⌉ comparisons instead of up to k−1. The total comparison count becomes \sum_{k=2}^{n} ⌈log₂(k)⌉ = Θ(n \log n) — matching merge sort's performance while keeping insertion sort's structure.

HW — Due Wednesday April 8

Problem: Prove that the 12-coin problem cannot be solved with 14 coins, and cannot even be solved with 13 coins.

14 coins: 2 × 14 = 28 > 27 = 3³. The decision tree has too few leaves. Impossible by counting. ∎

13 coins: Counting alone doesn't rule it out (26 ≤ 27). But for any first weighing with k coins per side, the three branches have sizes 2k, 2k, 2(13−2k). For all three to be ≤ 9: need 2k ≤ 9 AND 2(13−2k) ≤ 9, giving k ≤ 4 AND k ≥ 4.25. No integer k satisfies both. ∎

HW — Due Friday April 10

Problem: Given sequence x₀,x₁,x₂,x₃,x₄ with positive sum and indices mod 5. An operation selects a negative term xₖ, replaces it with −xₖ, adds xₖ to both neighbors. The doubly-infinite sequence sₙ satisfies sₙ₊₁ = sₙ + xₙ mod 5. What does the operation do to this infinite sequence?

Answer: The operation corresponds to a reflection of the infinite sequence: if xₖ is replaced by −xₖ, the infinite sequence transforms as sₙ \mapsto s_* - sₙ for some constant s_* — i.e., the sequence is reflected about a horizontal axis. The sum \sum xᵢ is preserved by each operation (since -xₖ + xₖ + xₖ = xₖ in the net effect on the sum), keeping the overall drift of the infinite sequence positive.
